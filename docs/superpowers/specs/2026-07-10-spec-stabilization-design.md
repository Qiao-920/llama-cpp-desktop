# llama.cpp Desktop 规格稳定化设计

## 背景

`spec/` 定义了 0.6.x 稳定化阶段的运行配置、聊天请求、附件、终端日志和发布要求。当前实现已经覆盖大部分界面和主流程，但关键策略散落在 `desktop/main.mjs` 与 `renderer/app.js` 中，缺少自动化测试，导致“界面看起来正常”与“最终请求或发布资产真实正确”之间没有稳定证据链。

本轮按用户授权同时覆盖三个交付方向：聊天安全链路、运行时与发布闭环，以及其余附件和日志规格缺口。

## 方案比较

### 方案 A：直接在两个入口文件中逐点修补

改动最少，但策略仍不可独立测试，重复请求构造和日志过滤会继续分叉。短期快，后续回归风险最高。

### 方案 B：大规模重构为完整分层架构

长期边界最好，但会扩大 0.6.x 稳定化的改动面，并让 UI、IPC 和运行时同时迁移。与本轮“修复并给证据”的目标不匹配。

### 方案 C：提取高风险纯逻辑，保留现有 Electron 结构

这是选定方案。只把运行参数策略、聊天消息整理、流式片段解析和日志过滤提取为无 Electron 依赖的模块，使用 Node 内置测试覆盖；`main.mjs`、`preload.cjs` 和 `app.js` 保留现有职责和交互方式。它能建立证据链，同时控制重构范围。

## 架构

### 测试底座

- 使用 Node 22 内置的 `node:test` 与 `node:assert/strict`，不增加生产依赖。
- `npm test` 运行全部 `test/*.test.mjs`。
- 每项行为改动先增加失败测试，再实现最小修改。

### 运行配置策略

- 新建 `desktop/lib/runtime-policy.mjs`，负责默认 host、风险提示、核心参数冲突检测和服务 URL 计算。
- 默认 host 改为 `127.0.0.1`。
- `0.0.0.0`、`ctx_size > 65536`、`request_timeout_ms < 30000` 返回结构化警告，由设置页直接展示。
- `extra_args` 不允许覆盖由界面管理的核心参数；保存或启动时给出用户可读错误。
- 状态区区分监听地址与本机可访问 API 地址，避免把 `0.0.0.0` 静默显示成 `127.0.0.1`。

### 聊天安全链路

- 新建 `desktop/lib/chat-pipeline.mjs`，统一规范化 `chat_template_kwargs`、合并 system 消息、排除本地消息并解析流式片段。
- 非流式与流式 IPC 共用同一个消息整理函数；最终最多一个 system，且只能位于索引 0。
- 流式片段同时识别 `content`、`reasoning_content` 和 `thinking`。
- 每次流式请求携带 `requestId`。主进程使用 `AbortController` 注册表取消指定请求，preload 暴露取消方法，生成中发送按钮变为明确的停止按钮。
- 超时、取消或服务错误后，已收到的部分 assistant 消息保留用于用户查看，但标记为 `localOnly`/失败状态，下一轮请求必须排除。

### 附件与输入

- 菜单固定为图片、音频、文本、PDF、系统消息、MCP 六项；删除当前阶段非目标的视频入口。
- MCP 显示禁用态和“暂未实现”原因；音频可选取，但明确说明当前只作为文件说明发送，是否理解取决于模型能力。
- 菜单使用视口最大高度和滚动兜底，不依赖固定的错误高度估算。
- 文件读取错误和能力警告在附件 pill 内可见；图片继续遵守 `min(280px, 52vw)` 与 `320px` 上限。

### 终端日志

- 新建 `desktop/lib/log-pipeline.mjs`，把清洗、分类、过滤和截断变成纯函数。
- 保留模型加载、CPU/CUDA/Metal、监听地址和错误；过滤 streaming JSON、prompt、HTML/CSS/JS 回显和空闲轮询。
- 主进程维护 `filtered`、`truncated`、`dropped` 计数，终端页最多展示 520 行，并分别说明过滤与容量截断数量。
- 原始日志不写入磁盘，也不进入普通 UI；自动测试以 fixture 证明输入到输出的过滤行为。

### 发布合同

- `.gitignore` 忽略 `dist-*`。
- Release workflow 在构建前验证 tag 严格符合 `vX.Y.Z`，且与 `package.json.version` 完全一致。
- 保留 exe 与 sha256 资产，启用自动生成发布说明并附加下载、已知限制和配置迁移说明。
- 不创建 tag、不推送 GitHub；本轮只验证本地合同与打包产物。

## 数据流

1. renderer 读取配置与警告，展示设置和运行状态。
2. renderer 构造带 `requestId` 的消息请求，main 使用统一聊天管线生成最终 OpenAI 请求体。
3. main 将 SSE 片段规范化为正文、思考和原始片段事件；renderer 更新同一条 assistant 消息。
4. 取消或失败时 main 终止请求，renderer 将部分回复标记为本地失败，不再进入下一轮。
5. llama-server 日志先经过纯过滤管线，再连同统计计数发送给 renderer。

## 错误处理

- 配置冲突在启动前失败，错误指明冲突参数。
- 端口、模型路径、超时和取消均转成短中文提示；完整内部内容只进入日志或原始输出。
- 附件读取失败仍保留一个错误 pill，用户可以移除后继续发送。
- 取消不显示为服务故障，也不污染历史请求上下文。

## 验收

- 自动：`npm test`、两个 `node --check`、`git diff --check`、Release 合同测试。
- 构建：临时输出目录打包，校验 exe 大于 10 MB，并生成/核对 SHA256。
- UI：在 `1365x768` 与 `1920x1080` 验证设置警告、附件菜单、生成中停止态、终端页和长回答底部。
- 运行时依赖真实 llama.cpp/模型的项目如 Qwen 连续对话、端口占用和 mmproj 视觉理解，记录为需用户本机模型配合的手工验收，不伪造通过证据。

## 非目标

- 不内置 llama.cpp、模型或 mmproj。
- 不实现视频理解、完整 MCP、macOS/Linux 打包。
- 不为所有模型猜测模板参数。
