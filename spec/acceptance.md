# 验收标准

所有验收项必须能被命令、截图、日志或代码位置证明。不能只写“看起来正常”。

## A. 基础质量

| ID | 验收项 | 通过标准 | 判定方法 | 必须证据 |
| --- | --- | --- | --- | --- |
| A-01 | Git 状态可解释 | 除开发者明确知道的未提交文件外，没有误删、误改、未知大文件。 | `git status -sb` | 命令输出。 |
| A-02 | 主进程语法正确 | `desktop/main.mjs` 语法检查退出码为 0。 | `node --check desktop/main.mjs` | 命令输出。 |
| A-03 | 渲染脚本语法正确 | `renderer/app.js` 语法检查退出码为 0。 | `node --check renderer/app.js` | 命令输出。 |
| A-04 | diff 无尾随空白 | 没有尾随空格、冲突标记、非法空白。 | `git diff --check` | 命令输出。 |
| A-05 | 可打包 | 生成 Windows exe，文件大小大于 10 MB。 | `npm run dist` 或 electron-builder | 产物路径、大小、sha256。 |

## B. 运行配置

| ID | 验收项 | 通过标准 | 判定方法 | 必须证据 |
| --- | --- | --- | --- | --- |
| B-01 | 默认上下文大小安全 | 新配置默认 `ctx_size` 为 `32768`。 | 查看默认配置和设置页。 | 代码位置或设置截图。 |
| B-02 | 高上下文警告 | 当 `ctx_size > 65536` 时，用户能看到内存风险提示。 | 设置页手动输入。 | 截图。 |
| B-03 | 请求超时足够长 | 默认 `request_timeout_ms` 为 `600000` ms，允许用户配置，但低于 `30000` ms 时必须有明确风险。 | 查看配置和请求逻辑。 | 代码位置或配置截图。 |
| B-04 | host/port 展示准确 | 服务地址与设置中的 host/port 一致，例如 `http://127.0.0.1:8080`。 | 启动服务后看状态栏。 | 截图。 |
| B-05 | 端口占用错误可读 | 端口被占用时，界面提示能指导用户检查端口或换端口。 | 模拟端口占用。 | 错误截图。 |

## C. 思考模式与聊天请求

| ID | 验收项 | 通过标准 | 判定方法 | 必须证据 |
| --- | --- | --- | --- | --- |
| C-01 | 支持 JSON 写法 | `{"enable_thinking":false}` 能作为 `chat_template_kwargs` 生效。 | 发送请求并检查请求体/日志。 | 请求摘要或日志。 |
| C-02 | 支持 CLI 写法 | `--chat-template-kwargs '{\"enable_thinking\":false}'` 能被规范化为 JSON 对象。 | 在设置中填入 CLI 写法后发送。 | 设置截图和请求摘要。 |
| C-03 | 启动参数同步 | 配置的模板 kwargs 同时用于启动参数和聊天请求。 | 查看启动命令和请求体。 | 日志摘要。 |
| C-04 | 系统消息在开头 | 任何请求最终 `messages[0].role` 必须是 `system` 或无 system；不得出现 system 在中间。 | 构造有历史、错误、本地消息的对话。 | 请求摘要。 |
| C-05 | Jinja 错误不再出现 | 正常聊天不出现 `System message must be at the beginning`。 | 使用 Qwen 模板发送连续对话。 | 聊天截图或日志。 |
| C-06 | 本地错误不污染上下文 | 超时、500、取消等 UI 错误不进入下一轮模型 messages。 | 触发错误后再发送一条消息。 | 下一轮请求摘要。 |
| C-07 | 思考区可折叠 | 若模型返回 `<think>` 或 thinking 内容，主回答与思考区分离，思考区可折叠。 | 使用会输出思考的模型。 | 截图。 |
| C-08 | 原始输出可打开 | 生成中和完成后，原始输出按钮都能点击；不能被空白容器挡住。 | 生成中点击原始输出。 | 生成中截图。 |
| C-09 | 最底部跟随 | 生成中默认跟随最新内容，用户未主动上滑时应保持可见底部。 | 长回答生成。 | 过程截图或录屏。 |

| ID | 验收项 | 通过标准 | 判定方法 | 必须证据 |
| --- | --- | --- | --- | --- |

| D-02 | 菜单项可点击 | 启用项点击后能打开文件选择或进入对应流程；禁用项必须有说明。 | 逐项点击。 | 测试记录。 |

| D-04 | 图片像聊天附件 | 图片显示为轻量附件气泡，不包进巨大空白卡片。 | 上传图片并发送。 | 截图。 |
| D-05 | 普通文件紧凑展示 | 文本/PDF/音频文件显示文件名、类型、大小，不生成大面积空白预览。 | 上传三类文件。 | 截图。 |
| D-06 | 文件名可读 | 长文件名必须截断显示，不能挤出容器。 | 上传长文件名文件。 | 截图。 |
| D-07 | 输入框不遮挡消息 | 最后一条回复和元信息不会被输入框覆盖。 | 长回答结束后看底部。 | 截图。 |

## F. 发布

| ID | 验收项 | 通过标准 | 判定方法 | 必须证据 |
| --- | --- | --- | --- | --- |
| F-01 | package 版本正确 | `package.json` 的 `version` 与发布 tag 一致。 | 查看文件和 tag。 | 文件片段和 `git tag`。 |
| F-02 | Release 标题正确 | GitHub Release 标题包含同一版本号。 | 打开 Release 页面。 | 链接或截图。 |
| F-03 | exe 资产存在 | Release 包含 `Llama.cpp-Desktop.exe`。 | 查看 Release 资产。 | 链接或截图。 |
| F-04 | sha256 资产存在 | Release 包含 `Llama.cpp-Desktop.exe.sha256`。 | 查看 Release 资产。 | 链接或截图。 |
| F-05 | 不上传构建垃圾 | `dist-*`、模型文件、node_modules 不进入提交。 | `git status -sb` 和 `.gitignore`。 | 命令输出。 |

## G. 失败不能蒙混的判定

以下情况直接判定为未通过：

1. 只说“修好了”，没有命令输出或截图。
2. 只在大窗口测试，未测试 `1365x768`。
3. 只测试完成态，未测试生成中态。
4. 只测试无附件聊天，未测试图片和普通文件。
5. 发布后没有校验 GitHub Release 资产。
6. 把本地错误消息继续发进下一轮模型上下文。
7. 终端日志仍默认展示完整 prompt 或流式 JSON。
