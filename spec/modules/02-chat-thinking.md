# 模块规格：聊天与思考模式

## 模块目标

保证聊天请求稳定、模板参数真实生效、思考内容可控展示，并且错误不会污染下一轮上下文。

## 请求管线

```text
用户输入
  -> 附件归一化
  -> 历史消息过滤
  -> system 消息合并到最前
  -> chat_template_kwargs 归一化
  -> /v1/chat/completions
  -> 流式内容解析
  -> thinking / answer / raw output 分区展示
```

## 思考模式规则

| 输入形式 | 期望处理 |
| --- | --- |
| `{"enable_thinking":false}` | 直接作为 JSON 对象使用。 |
| `--chat-template-kwargs '{\"enable_thinking\":false}'` | 提取并反转义为 JSON 对象。 |
| 空字符串 | 不发送该字段。 |
| 非法 JSON | UI 必须提示配置错误，不应静默吞掉。 |

## 系统消息规则

1. 如果存在 system 消息，最终请求中必须只有一个合并后的 system 消息。
2. system 消息必须在 `messages[0]`。
3. 历史消息中的 system 片段不得散落在中间。
4. 本地 UI 错误、状态提示、超时提示不得作为 assistant 内容进入下一轮。

## 展示规则

| 内容 | 展示方式 |
| --- | --- |
| 普通回答 | 主回答区域直接展示。 |
| `<think>...</think>` | 折叠到思考区，默认行为由设置控制。 |
| 原始输出 | 放入“原始输出”折叠区，生成中也可点击。 |
| 错误 | 系统错误气泡，文案面向用户；长堆栈进入终端日志。 |
| token/耗时/速度 | 元信息 pill，不能遮挡正文。 |

## 常见失败模式

| 失败 | 判定 |
| --- | --- |
| 关闭思考后仍输出大量思考 | 检查 kwargs 是否真的发到启动和请求。 |
| Jinja system message 报错 | 检查 system 是否被合并到开头。 |
| 回答内容消失 | 检查 `<think>` 解析是否误吞正文。 |
| 原始输出点不开 | 检查生成中状态是否有透明遮罩或高度为 0 容器。 |
| 下一轮带入错误 | 检查错误消息是否标记为本地消息并从请求过滤。 |
